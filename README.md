어떤 기술 스택을 왜 선택했는지
-> 사실상 전부 최근 가장 많이 사용하는 스택을 골랐다.
-> 왜냐? 많이 쓰는 스택이 노출된 취약점도 많을 것이기 때문이다.

프론트 - React (Vite)
이유 
- 가장 많이 사용하는 스택을 선택함.
- 평소 개발하는 스택을 선택함.
- 담에 next js 도 써볼까 싶음

백 - node js
- 프론트와 동일하게 가장 많이 사용하고 평소 사용하는 스택을 선택함.
- 프론트엔드와 동일한 JS 를 사용하여 LLM 에게 개발 시킬 경우 분석이 비교적 쉬움.

디비 - SQLite
- 작은 웹사이트를 개발하는거였으니 빠른 배포를 위해 채택.
- 표준 SQL 문을 그대로 사용하여(=다른 디비와 크게 다르지 않음) 지금 다른 디비를 사용할 이유가 없음.
- 간단한 조사 결과 SQLite만의 취약점도 있으니 스택별 취약점 차이 분석시에도 사용 가능.

리버스프록시 - nginx
- 리버스프록시 쓸 때 가장 많이 쓰이며, 요구 사항에 있었으니 사용함.
- HTTPS 연결할 때 Cloudflare 쓸지, Let's Encrypt 쓸지 고민했는데 
    많이 쓰는 것은 Cloudflare임에도 Let's Encrypt 가 자유도 및 공부할 때 도움되리라 생각하여 Let's Encrypt로 결정함.
- 나중에 HTTPS 관련 학습을 진행한다면 Let's Encrypt 위에 Cloudflare도 붙여서 쓰지 않을까 생각함.

배포 - 홈서버 + docker
- 아오 여기서 해맸다.
- 코드는 잘 짰음
- 클라우드플레어에 도메인-ip 연결하고(여기서 사서 그럼)
    a 레코드--로 설정
- dockge로 도커 이미지 돌리고- Frontend, Backend, Nginx 3개 컨테이커로 구성- 8080으로 묶음, 도커 볼륨 설정 추가해서 디비 안 날라가게 함
- NPM(리버스프록시) 에서 SSL 인증서 받아서 적용함, ip - domain 도 연결함
    외부 접속 -> 클라우드플레어(ip-domain 매칭) -> NPM(리버스프록시)(tiny-mp.com 를 내부8080으로 연결) -> 8080에서 nginx(프록시)로 연결 -> 프록시가 Frontend, Backend, DB 로 접속 분배함

전체 구조 (프론트 / 백 / DB / 프록시)

>>프론트(React)
- 화면 보여줌(로그인창), 유저 정보 입력 후 버튼 누르면 프록시로 정보 전달(HTTPS)
- index html에서 main jsx 호출하며 시작함
- main jsx 에서 app jsx 불러와서 서비스 띄우고
- index css 에서 디자인 소스 불러와서 디자인 적용함
>>nginx 리버스 프록시(nginx, VIte)
- 프론트에서 받은 정보를 nginx가 받아 백엔드로 전달, 백엔드에서 받은 정보를 nginx가 받아  프론트로 전달, HTTPS 통신
- Vite는 프론트가 CORS 안뜨게 하는 등 기능인데 개발할 때만 씀(처음 알았네)
>>백(node js)
- 프록시에서 받은 정보를 바탕으로 디비에 저장하거나 조회 후 로직에 따라 프록시로 정보 반환
- server.js 단일 파일로 구현하고 CORS 등 취약점 그냥 둠
- 기본 미들웨어에서 json 파싱, CORS 풀고, jwt 관련 미들웨어는 프로필 엔드포인트/profile 에서만 요구함
- 엔드포인트는 /login, /register, /profile 있음
>>디비(SQLite)
- 백엔드의 요청을 받아 정보를 저장하거나 반환
- 백엔드에서 서버 실행될 때 디비 스키마를 만든다. db.run(`CREAT~~
- 나중에 sql 파일로 빼거나 Prisma 같은걸로 정의하게 해야겠는데 지금은 필요 없을듯

막혔던 부분과 해결 방법

>>LLM 없으면 귀찮았을텐데 딸깍으로 수월하게 진행함
- 다만 코드 한번 쭉 뜯어보고 주석 달면서 이해하는게 어색해서 시간 좀 걸림
- 코드 보는 연습 웹해킹에는 필요할테니 잘 한거라고 생각함


>>jwt 가 뭐냐
- 대충 세션 유지용으로 서버가 사용자에게 배포하는 토큰으로 알고 있었음
- base64로 인코딩된 헤더, 페이로드, 시그니처로 구성되어있음
- 이름과 같이 json 으로 준다
- 시그니처는 서버가 가진 비밀키로 암호화되어있어서 위변조가 불가능함 -> 노출되면 귀찮아짐
- jwt 는 확인 가능 수정 불가 구조임 -> 비밀키로 서명되었기 때문
- 프로그램의 경우 1h 제한을 걸어둠

>>Node js 개발할줄 몰랐음
- 주석 달면서 뜯어봐서 코드 리뷰정도는 할 수 있을듯 이제

>>CORS 설정 풀어놓아서 해킹에 취약한 상태임
- 일부러 그렇게 해둠
""
//크로스오리진 풀어줌. 그 뭐냐 프론트에서 요청 받아야하니까
app.use(cors());
""
- 나중에 아래와 같이 고칠거임. 왜 안고쳤냐고? 이거 뚫는 실습 하면 좋겠다는 바람에. CORS 우회 관련.
""
const corsOptions = {
    origin: '~~',
    optionsSuccessStatus: 200

""
